# Changelog

## Step 1: The Memento Contract — In-Memory Only

- **Date**: 2026-02-27
- **What I built**: A pure in-memory key-value store implementing the `Memento` API contract, backed by a JavaScript `Map` (`this._cache`), featuring synchronous reads and asynchronous (`Thenable`) writes.
- **What I learned**: The asymmetry between `get()` and `update()` is a structural necessity, not just a preference. To prevent the single-threaded Node.js Extension Host from freezing during slow disk reads, the system is forced to maintain a complete, authoritative in-memory `Map` (`this._cache`) so `get()` can return data synchronously. When `update()` is called, it immediately mutates this cache so subsequent reads are instantly accurate, but it must return a `Promise` because the expensive O(N) serialization and file writing must be delegated to background OS threads.
- **Model correction**: N/A — The in-memory map matches the expected behavior.

## Step 2: File-Backed Persistence — Naive Immediate Writes

- **Date**: 2026-03-02
- **What I built**: A `StorageBackend` using Node's built-in `fs` module that wires directly to the `Memento` cache. Every `update()` call triggers an immediate `JSON.stringify()` serialization and a full asynchronous file overwrite. I also implemented a `writeCount` tracker to explicitly observe the physical I/O cost.
- **What I learned**: Naive immediate writes expose the severe physical friction between variable-length data (JSON) and contiguous file storage. Because a file is a contiguous stream of bytes , changing a value to be longer or shorter means you cannot simply update those specific bytes in place—doing so would overwrite adjacent JSON syntax (like quotes or braces) and corrupt the file. Unlike SQL database engines that use slotted pages and pointers to manage variable-length data in place , standard file system APIs lack an "insert and shift" capability. Consequently, the only safe way to mutate a JSON file is to rewrite the entire thing from scratch. Observing five sequential, tiny key updates trigger five complete, full-file disk overwrites proved that this O(N) penalty causes catastrophic I/O amplification.
- **Model correction**: N/A — This physical constraint proves the initial hypothesis: without an application-layer write queue (debouncing) to batch these operations, the system is fundamentally unsustainable.

## Step 3: Multi-Extension Namespace Isolation

- **Date**: 2026-03-02
- **What I built**: An `ExtensionHostStorage` class that acts as a central factory for multiple `Memento` instances, mapping each extension to its own isolated branch in a nested JSON tree (e.g., `store[this._namespace] = data`). I also built a rigorous test harness to explicitly observe the physical I/O costs and stress-test the system with uncoordinated, concurrent updates.
- **What I learned**: While the nested tree perfectly solves the local problem of key collisions, the decentralized implementation exposed a fatal architectural flaw. I initially masked this issue in my test harness by meticulously chaining `await` on every single `update()` call, effectively forcing the engine to serialize the disk writes. However, this consumer-level serialization is impossible in a production environment. Independent extensions do not share a control flow; they cannot coordinate their `await` calls. When multiple extensions react to the same system event concurrently, the Node.js event loop happily dispatches their uncoordinated async file commands simultaneously. Handing this concurrency off to the non-deterministic OS thread scheduler results in two catastrophic failure modes :
    - **Store Corruption (Mid-Flight Collision)**: If Extension B calls `fs.readFileSync` at the exact microsecond the OS has truncated the file for Extension A's `fs.writeFile`, B reads an empty or partially written file. This instantly crashes the system with a JSON `SyntaxError`.
    - **Silent Data Loss (Ghost Writes)**: If both extensions read the same stale file state before either write completes, they both mutate their isolated branches and race to overwrite the entire file. The last writer completely obliterates the first writer's data, and because the OS executes these writes independently of JavaScript errors, the destruction is permanent.
- **Model correction**: The nested object tree hypothesis is structurally correct, but allowing individual `Memento` instances to independently manage disk synchronization is structurally bankrupt. Relying on consumer-level `await` is a leaky abstraction that fails under real-world concurrency. To prevent OS-level race conditions, the system must maintain a centralized "source of truth" in memory at the host level, completely separating synchronous state mutations from a centralized, debounced disk queue.

## Step 4: Debounced Write Batching and the `update()` Promise

- **Date**: 2026-03-03
- **What I built**: A centralized `ExtensionHostStorage` that maintains a single, authoritative in-memory state (`this._store`) for all extensions. I completely severed the `Memento` from the disk; it now synchronously pushes state snapshots up to the host. I also built a `FlushScheduler` that uses a debounced queue (`setTimeout`) to batch these updates, collecting individual `Promise` resolvers in an array (`this._pendingResolvers`).
- **What I learned**: Centralizing the state completely eliminates the fatal read-modify-write race condition . Because all concurrent extensions instantly mutate the same shared in-memory object, the OS-level thread scheduler is removed from the equation. The debounce mechanism solves the I/O bottleneck: firing 10 rapid updates now physically overwrites the file only once instead of 10 times, drastically reducing the O(N) penalty. Crucially, this exposes the true semantics of the `update()` Promise: when an extension calls `await update()`, it is not waiting for its own exclusive disk operation. It is joining a batch and waiting for the scheduler to flush the entire queued state to disk.
- **Model correction**: My initial hypothesis predicted *both* a debounce mechanism and an internal priority queue to order state updates. While the debounce was proven structurally necessary, the priority queue hypothesis was completely falsified. Because the persistence layer resolves the variable-length data constraint via an atomic, full-store serialization (`JSON.stringify`), per-key ordering within a batch is physically meaningless. The entire tree hits the disk at once.

## Step 5: `setKeysForSync` — The Sync Registry

- **Date**: 2026-03-03
- **What I built**: Implemented `setKeysForSync(keys)` on the `Memento` class to maintain an in-memory `Set` of sync-eligible keys. I also built a `getSyncData()` method on the `ExtensionHostStorage` that iterates across all active namespaces and extracts a point-in-time snapshot of only the registered keys.
- **What I learned**: The registry acts purely as a declarative filter that lives entirely in memory. It keeps the sync domain logic completely isolated from the physical persistence layer. Because extensions inherently re-declare their sync keys during activation, there is zero need to persist this metadata to the disk, avoiding unnecessary structural bloat.
- **Model correction**: My initial hypothesis predicted two mechanisms: a separate registry and an event-driven sync queue (event sourcing). The registry was confirmed, but the event sourcing hypothesis was explicitly falsified. I initially rejected event sourcing purely on the basis of overhead, but analyzing the underlying mechanisms reveals a deeper structural truth: event sourcing solves the wrong problem. The fundamental precondition for event sourcing—state that is too large or too critical to snapshot—does not exist here. Because the entire sync-eligible state for an extension is just a handful of keys, a point-in-time snapshot (`getSyncData()`)  is not just cheaper, it is structurally superior. It guarantees absolute state consistency, completely eliminating the risk of partial-replay corruption inherent to event logs.
