# prereq-write-ahead-log

## Subject

Append-only sequential log with explicit byte-offset tracking, serving as both a crash-recovery mechanism and an offset-addressable entry store — the structural answer to the fundamental unsafety of in-place file mutation.

## Core Question

What makes sequential append to a log file both safer under crash and faster under load than in-place mutation of a data file, what is the minimum mechanism required to recover consistent state from such a log after unexpected process termination, and why does the append-only constraint make byte offsets a sufficient addressing scheme — eliminating the need for a separate index to locate entries?

## Hypothesis

Databases store data in structured formats — B+ trees, hash tables, heap files — that require in-place modification. An `UPDATE` must find the specific page containing the target row and rewrite bytes inside it. If the process crashes mid-rewrite — while a page is half-written, or between writing a leaf page and its parent pointer — the data structure is left in a structurally corrupt state. The tree's invariants are broken. The data cannot be trusted.

The WAL resolves this through a single rule: before modifying any data in place, first append a record describing the intended modification to a sequential, append-only log file; only after that record is durable on the storage device, perform the actual in-place modification. If a crash occurs before the log record is durable, the modification never happened and the data is untouched. If a crash occurs after the log record is durable but before the in-place modification completes, recovery replays the log and finishes the operation. In every case, the system recovers to a consistent state.

I believe this protocol works because the append-only constraint is a single structural decision from which crash safety, sequential I/O performance, and zero-cost offset addressing all derive as direct mechanical consequences — not as independent features bolted together, but as geometry inherent to the structure.

Concretely, I expect the following mechanical behaviors:

- **The Three Layers Beneath a File Write:** A file is not "bytes on disk." It is a metadata record (an *inode* on Unix) that points to data blocks, and those data blocks are accessed through the OS page cache — a volatile RAM buffer managed by the kernel. When Node.js calls `fs.writeSync()`, the bytes travel to the page cache, not to the storage device. The kernel marks those pages as *dirty* and flushes them to the physical medium in the background, triggered by time thresholds (`dirty_expire_centisecs`), memory pressure (`dirty_background_ratio`), or an explicit `fsync()` call. This three-layer architecture — inode metadata, page cache, storage hardware — means that a successful `write()` guarantees only that subsequent `read()` calls will see the data (because they also read from the page cache). It guarantees nothing about survival across power loss. The durability gap between "in RAM" and "on storage" is the central tension the WAL must navigate.

  The storage hardware itself introduces further complexity. On an NVMe SSD, you cannot overwrite a NAND flash cell in place — you must erase an entire block (256KB–4MB) to write fresh data. The SSD's firmware (the Flash Translation Layer, or FTL) hides this by appending new writes to free pages in an active erase block and updating an internal mapping table, marking the old pages as stale. A sustained random write workload drowns the FTL in garbage collection — the firmware must read fragmented blocks, condense surviving pages, and erase old blocks — consuming bandwidth that could otherwise serve new writes. Sequential writes avoid this entirely: they fill erase blocks cleanly, enabling the NVMe controller to saturate its deep I/O submission queues (up to 64K commands) and parallelize across multiple flash channels without GC stalls. The SSD itself survives by appending and garbage-collecting — the same log-structured pattern the WAL executes in software.

  One layer deeper: modern filesystems like ext4 solve their own crash-recovery problem (a crash between writing a data block and updating the inode that points to it) using a *journal* — the filesystem's own write-ahead log. `fsync()` is reliable precisely because the filesystem journals its metadata updates before committing them. Our WAL sits one layer above this: our durability guarantees rest on the filesystem's own crash-recovery mechanism.

- **Torn-Write Detection via Length Framing:** If the process dies mid-append, the WAL file will end with an incomplete record. By framing every entry with a 4-byte unsigned integer (big-endian `u32`) that declares the exact payload length, the recovery parser can detect two distinct failure modes. First, if the parser cannot read a complete 4-byte length header before hitting EOF, the length frame itself was torn — it discards the partial bytes. Second, if the parser reads a complete length header but hits EOF before consuming the promised number of payload bytes, the payload was torn — it discards the header and the partial payload. In both cases, the parser truncates the file to the end of the last complete entry using `fs.ftruncateSync()`, restoring the log to a consistent state.

  Length-prefix framing is chosen over line-delimited text (e.g., one JSON object per `\n`) for two reasons. First, a delimiter-based scheme cannot distinguish "record ends here" from "record was truncated here" without attempting to parse the content — which is expensive and ambiguous at certain truncation points. The length prefix makes this determination a simple integer comparison: either the promised bytes are present or they are not. Second, if the payload can contain the delimiter byte (a newline inside a JSON value), the framing breaks entirely, requiring escaping that inflates size and complicates parsing. The length prefix is payload-agnostic — it frames arbitrary binary content without restrictions.

- **The Vanished-Write Problem:** There is a third failure mode distinct from torn writes. If the process calls `write()` (which copies data into the OS page cache) but crashes before calling `fsync()`, the entry may never reach the physical storage device at all. In this case, the file on disk ends cleanly at a prior entry boundary — there is no torn record to detect. The entry simply does not exist. This means the parser will report a consistent log, but one that is missing the last committed operation. Any operation that the application treated as "successful" before the fsync boundary is, in reality, a lie. The torn-write failure modes (1 and 2) are *structural* problems solvable by the framing protocol. The vanished-write failure mode (3) is a *durability* problem solvable only by `fsync()`. This is why the WAL's commit protocol is: append the entry, fsync, then acknowledge success. Any acknowledgment before fsync is a durability gamble.

- **The `fsync` Durability Boundary:** `fs.fsyncSync(fd)` issues a command to the kernel: flush every dirty page associated with this file descriptor to the storage device and do not return until the device confirms durability. The true cost is paid in hardware physics: the kernel issues write commands to the NVMe controller, follows them with an explicit hardware cache-flush command (`Flush` on NVMe, `FLUSH CACHE EXT` on SATA), and the controller halts — forcing the volatile data in its onboard DRAM cache into NAND flash cells and blocking until the electrical charge is verified stable. Only then does the syscall return. An `fsync()` on NVMe takes roughly 0.02–0.5ms — small in absolute terms, but it establishes an inescapable electrical ceiling: maximum durable writes per second on NVMe ≈ 2,000–50,000 without batching. Node.js also exposes `fs.fdatasyncSync(fd)`, which flushes file data but skips metadata updates (size, timestamps) unless required for correct data retrieval. For an append-only log where the file only grows, `fdatasync` is strictly cheaper. PostgreSQL uses `fdatasync` by default (`wal_sync_method = fdatasync`).

  Every persistence system chooses a position on a durability spectrum where durability and throughput move in opposite directions. There are three canonical positions: **no fsync** (maximum throughput, all unflushed data at risk — Redis `appendfsync=no`), **fsync per batch / group commit** (amortized cost, bounded data loss of one batch window — PostgreSQL's `commit_delay`, Redis `appendfsync=everysec`), and **fsync per write** (zero data loss, throughput capped at device fsync latency — Redis `appendfsync=always`). Our étude builds the per-write position to expose the raw durability cost; the group commit and no-fsync positions are excluded from scope but their necessity is a direct consequence of the cost we will observe.

- **Sequential vs. Random I/O:** Sequential appends to the end of a log file will outperform in-place updates to a data file on every storage medium, for four reinforcing reasons: hardware prefetching (both HDDs and SSDs detect sequential patterns and speculatively read ahead — random access defeats this because the hardware cannot predict the next position), OS read-ahead (the kernel's page cache issues larger I/O requests when it detects sequential access, amortizing per-request overhead), device command queuing (NVMe SSDs merge and reorder sequential commands for maximum internal parallelism — random small writes fragment the queue), and CPU cache locality (sequential access to a `Buffer` hits L1/L2 cache lines efficiently — random access causes cache misses).

  The gap is structural, not incidental. A premium NVMe SSD rated at 5,000 MB/s sequential write throughput may deliver only ~200 MB/s effective bandwidth under random 4KB writes (~50,000 IOPS) — roughly a 25x gap — because the FTL spends its compute on garbage collection rather than accepting new data. Any system that converts random I/O into sequential I/O captures this entire gap. The WAL is precisely such a system.

  However, I suspect that under our per-entry fsync regime, the fsync latency will dominate the I/O pattern advantage. Each fsync forces a hardware round-trip, which means the sequential advantage may be real but invisible at our measurement granularity. This is exactly the design pressure that drives production systems toward group commit — amortizing the durability cost across many entries to let the sequential throughput advantage re-emerge.

- **The Catastrophe of In-Place Mutation:** Even for a simple key-value store, in-place mutation of a data file is structurally unsafe and prohibitively expensive — doubly bankrupt. The first failure is the **O(N) rewrite penalty**. Consider a JSON file storing `{"name":"Al"}` — 13 bytes. Update the value to `"Alexander"` and the new content is 21 bytes. You cannot insert 8 bytes into the middle of a file — files are contiguous byte sequences backed by fixed-size blocks. Inserting would require shifting every subsequent byte forward, rewriting every subsequent block. The only option is to rewrite the entire file from byte 0. A 1-byte change to a 100MB file costs 100MB of I/O — O(N) where N is the total file size. The second failure is the **crash corruption hole**. `fs.writeFileSync(path, data)` internally truncates the file to 0 bytes, then writes the new content. If the process crashes between truncation and completion, the file contains neither the old data nor the new data — it is empty or partially written, the state irrecoverably corrupt. Even the safer pattern (write to a temp file, atomically rename) cannot recover the *sequence of operations* that led to the current state: if "debit A by $100, credit B by $100" crashes after the debit but before the credit, the money has vanished, and in-place mutation provides no mechanism to detect or replay the incomplete operation.

  The WAL solves both simultaneously through the append-only invariant: it never modifies existing bytes, only appends new ones. The in-memory `Map` is the "current state," and the log is the recipe for reconstructing it. The variable-length rewrite problem does not arise because the log never rewrites. The crash corruption problem does not arise because old data is never destroyed.

- **Offset Stability as Zero-Cost Addressing:** In an append-only file, once an entry is written at byte offset X, that entry will be at byte offset X for the lifetime of the file. No bytes before it will ever be inserted, deleted, or resized. This makes the byte offset a permanent, content-independent identifier — the equivalent of a primary key — generated for free at write time (it is simply the file's length before the append). In a mutable file, any insertion or deletion shifts subsequent bytes, invalidating all previously-recorded offsets and necessitating a separate index (e.g., a B-tree) to map logical identifiers to physical positions. The append-only log needs no such index. The caller receives the byte offset from `append()` and can retrieve that record at any future time with a single positional read at the known position — O(1), no traversal.

  This is the structural property that allows a downstream system like Kafka to use the byte offset as a message ID: consumers store a single integer (their current offset) and can resume consumption from any point in the log by seeking directly to that byte position. PostgreSQL's LSN (Log Sequence Number) is literally a byte offset into the WAL stream — `XLogRecPtr` is a 64-bit byte position, and locating a WAL record means seeking directly to the LSN with no index traversal.

  Production WALs often track the append offset manually in memory (initialized via `fs.fstatSync()` on startup) rather than relying on the OS `O_APPEND` flag, because the caller needs to know the exact byte offset *before* the write so it can be returned as the record's permanent address.

- **The Space Tradeoff:** The append-only design buys crash safety and addressability by going bankrupt on space. Every new write increases the file size. If a key is updated 1,000 times, all 1,000 versions exist in the log, though only the latest matters. This is a fundamental systemic compromise, not a bug. The broader system must solve space reclamation through log rotation (Kafka segments), checkpointing (PostgreSQL flushes dirty pages and records the checkpoint LSN, allowing older WAL segments to be recycled), or compaction (LSM trees discard superseded records). The WAL's responsibility is append and recover. Eviction is a downstream concern — excluded from this étude's scope, but its necessity is a direct consequence of the append-only invariant.

## Scope Boundary

This study intentionally excludes the following. Each exclusion is a simplification with known consequences.

**Log compaction / segmentation.** Production WALs do not grow forever. PostgreSQL creates checkpoints (`CreateCheckPoint()` in `xlog.c`) that allow old WAL segments to be recycled. Redis performs AOF rewrite (`rewriteAppendOnlyFile()` in `aof.c`). Kafka splits the log into time- or size-bounded segments (`LogSegment` in `log/LogSegment.scala`), deleting entire segment files in O(1) metadata operations when they exceed the retention window. Excluded because compaction addresses the space tradeoff — a direct consequence of the append-only invariant — which is orthogonal to the core append-and-replay mechanism this étude isolates. Consequence: our log grows without bound.

**Concurrent writers.** PostgreSQL serializes WAL insertions using `WALInsertLock`, an array of LWLocks managed in `xlog.c`. Kafka partitions are single-writer by design (only the partition leader appends). We use a single writer to isolate the sequential I/O mechanism from concurrency control. Consequence: no contention, no need for insertion locking.

**Group commit / batched fsync.** This étude builds the per-write fsync position on the durability spectrum to expose the raw cost of commitment. PostgreSQL occupies the group-commit position by default, batching multiple transactions' durability guarantees through `XLogFlush()` and the `commit_delay` GUC. Redis offers three positions: `appendfsync always` (per-write), `everysec` (per-batch), and `no` (no fsync). Kafka batches via `log.flush.interval.messages` and `log.flush.interval.ms`. Excluded because the throughput collapse we observe under per-write fsync IS the design pressure that makes group commit necessary — the étude exposes the disease so the cure is self-evident. Consequence: our throughput will be dramatically lower than production WALs.

**Log shipping / replication.** WAL records can be streamed to replicas (PostgreSQL streaming replication via `walsender.c`, FoundationDB's log server role, Kafka ISR replication). Excluded because replication is a distributed systems concern, not a local I/O mechanism.

**CRC-32 checksumming.** Production WALs append a CRC-32 checksum (4 bytes) after each record's payload to detect not just truncation but silent bit-level corruption — bit-rot, firmware bugs, electrical interference. CRC-32 is a non-cryptographic hash that maps arbitrary-length input to a 32-bit value with a ~1-in-4.3-billion false negative rate: fast (hardware-accelerated on modern CPUs via the `CRC32` instruction), deterministic, and sensitive (flipping a single input bit changes roughly half the output bits). PostgreSQL computes CRC-32C for each WAL record (`pg_crc32c` in `pg_crc32c.h`). Kafka computes CRC-32C per record batch (`DefaultRecordBatch` in `record/DefaultRecordBatch.java`). Redis AOF does not use per-entry checksums, relying instead on RESP syntax parsing — one reason AOF recovery is more fragile. We exclude CRC to keep the record format minimal and the recovery logic focused on the structural mechanism: length-prefix framing alone is sufficient for torn-write detection. The gap — inability to detect silent corruption where all bytes are present but some have been altered — is deliberate.

**Consumer offset persistence.** Kafka consumers persist their offsets externally (historically in ZooKeeper, now in the `__consumer_offsets` internal topic). Our WAL exposes the read-at-offset and scan-from-offset interface, but does not persist consumer cursors. That is a downstream concern for the `kafka-partitioning` étude.

## Prerequisites

None. This is a prerequisite étude (depth 0). All dependencies are inlined:

- **Key-value state**: A JavaScript `Map` serving as the in-memory state that the log protects. No invariants beyond get/set/delete. Inlined.

- **Binary entry encoding**: Length-prefixed records using Node.js `Buffer` — a fixed-length sequence of raw bytes with typed read/write methods, necessary because JavaScript strings are UTF-16 and do not expose individual bytes. Key operations: `Buffer.from(string, 'utf8')` converts a string to raw UTF-8 bytes; `Buffer.allocUnsafe(4)` allocates a 4-byte header without zero-initialization; `header.writeUInt32BE(length, 0)` writes the payload length as a 32-bit unsigned big-endian integer (big-endian is conventional network byte order — most significant byte first — matching PostgreSQL's WAL format and most binary protocols); `header.readUInt32BE(0)` reads it back; and `Buffer.concat([header, payload])` combines header and payload into a single contiguous record. The concat step is mechanically important: issuing two separate `writeSync()` calls (one for header, one for payload) would double the syscall overhead and, under concurrent access, risk interleaving bytes from different records. Concatenating in RAM first ensures a single atomic `writeSync()` call delivers the entire record to the page cache as one indivisible block. The 4-byte (32-bit) length header supports payloads up to ~4GB — 2 bytes (64KB max) would be restrictive, 8 bytes (64-bit) is unnecessary overhead. Inlined.

- **File I/O**: Node.js built-in `fs` module. The WAL uses six specific operations, each with a distinct role in the write path, read path, or recovery path. `fs.openSync(path, flags)` opens a file and returns a file descriptor — an integer handle the kernel assigns for all subsequent operations; flags control the mode (`'a+'` for read-write append, `'r'` for read-only, `'r+'` for read-write on existing file, `'w+'` for read-write with truncation). `fs.writeSync(fd, buffer, offset, length, position)` writes bytes at a specific byte position — a positional write equivalent to POSIX `pwrite()`, used instead of `appendFileSync` because the WAL must know the exact offset before writing so it can return it as the record's permanent address. `fs.readSync(fd, buffer, offset, length, position)` reads bytes from a specific position — a positional read equivalent to `pread()`, returning the number of bytes actually read; a return value less than `length` (a *short read*) is how the recovery parser detects torn writes. `fs.fsyncSync(fd)` forces dirty pages to the storage device as described in the hypothesis. `fs.fstatSync(fd)` returns file metadata including `stat.size` in bytes — used on startup to initialize the write offset so the WAL survives restarts without losing its position. `fs.ftruncateSync(fd, length)` truncates the file to exactly `length` bytes — the repair operation that discards torn tail records during crash recovery. No external dependency.

## Module Interface

This étude is a reusable building block. It must export a clean API via `module.exports` that downstream études (`lsm-tree`, `kafka-partitioning`) can `require()`. The interface must support:

- **`append(entry)`** → appends a length-prefixed record to the log via a single positional `writeSync` at the tracked offset, advances the offset, and returns the byte offset where the entry was written — the record's permanent address. This is the sequential write path.
- **`readAt(offset)`** → seeks to the given byte position via positional `readSync`, reads the length header, reads the payload, and returns `{ entry, nextOffset }`. This is the offset-stable random-access read — O(1), no index traversal. The `nextOffset` value enables sequential iteration from any starting point.
- **`scanFrom(offset, callback)`** → iterates entries sequentially from a given offset to EOF, invoking `callback(offset, entry)` for each complete entry. Returns the high-water mark (the offset immediately after the last entry read).
- **`replay(callback)`** → scans from offset 0, reconstructing state. Alias for `scanFrom(0, ...)` specialized for crash recovery. Calls `recoverLog` first to detect and truncate any torn tail records.

## Brand Mapping

The write-ahead rule — log the intent before performing the mutation, replay on crash — is the common protocol. Each system implements it with different tradeoffs:

- **Redis AOF** (Append Only File): `src/aof.c` — Redis logs every mutating command (`SET`, `LPUSH`, `ZADD`) as a RESP-formatted text string appended to the AOF file before the response is sent to the client. On restart, `loadAppendOnlyFiles()` replays the entire AOF sequentially to reconstruct the in-memory dataset from scratch. Corresponds to our replay-from-zero recovery path and our KV store application pattern (Step 7): log first, mutate in-memory state, then acknowledge. Redis AOF does not use per-entry checksums, relying on RESP syntax parsing for truncation detection.
- **PostgreSQL WAL** (Write-Ahead Log): `src/backend/access/transam/xlog.c` — Every `INSERT`, `UPDATE`, `DELETE` first writes a WAL record via `XLogInsertRecord()` describing the page modification. Only after `XLogFlush()` confirms the record is on disk does PostgreSQL modify the actual heap or index page in the shared buffer pool. Each record's Log Sequence Number (LSN) is its byte offset in the WAL stream — `XLogRecPtr` is a 64-bit byte position. On crash, `StartupXLOG()` replays unflushed WAL records from the last checkpoint's LSN to restore page consistency. Corresponds to both our offset-tracking mechanism and our crash recovery path.
- **Kafka Log**: `core/src/main/scala/kafka/log/LogSegment.scala` — Kafka appends record batches to segment files. Each batch has a base offset that serves as its permanent identifier. Consumers fetch by offset — the broker calls `FileRecords.searchForOffsetWithSize()` (in `log/FileRecords.java`) to translate the logical offset to a physical file position and reads directly from there, often via `sendfile()` zero-copy. Corresponds to our read-at-offset and scan-from-offset interface — the same append-only sequential I/O primitive that powers database crash recovery also powers the highest-throughput messaging system in production.
- **FoundationDB**: Uses a distributed WAL architecture where transaction logs are written to dedicated log server processes before being applied to the B-tree storage engine on storage servers. On recovery, it replays from the last durable snapshot — the same append-then-apply structure, distributed across dedicated processes.
